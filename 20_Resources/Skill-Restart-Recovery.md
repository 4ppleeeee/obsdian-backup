# OpenClaw 技巧：利用一次性 Cron 实现网关重启后的“断点续传”回复

## 痛点背景
在 OpenClaw 中，执行 `gateway restart` 或修改关键配置会导致 WebSocket 连接中断。如果此时 Agent 正在处理复杂任务，连接断开后 Agent 会处于“睡眠”状态，无法在网关上线后主动向用户报到，导致用户体验出现“断更”。

## 核心方案：一次性延时 Cron 任务 (One-shot Delayed Recovery)

该方案结合了 **文件持久化** 和 **系统调度器**，实现了物理级的主动恢复。

### 1. 逻辑设计 (纳入 global-rules)
- **重启前 (Pre-restart)**：
    - 检查是否有未完成的任务。
    - 将任务摘要写入 `memory/pending-restarts.json`。
    - **关键步骤**：通过 CLI 注册一个一次性 Cron 任务，设定在 N 分钟后执行（例如 3 分钟）。
    - 命令示例：`openclaw cron add --name "recovery" --at "+3m" --delete-after-run --message "执行恢复逻辑..."`
- **网关重启**：网关携带 Cron 计划重启。
- **重启后 (Post-restart)**：
    - 时间一到，Cron 引擎自动唤醒一个独立的 Agent 会话。
    - Agent 读取 JSON 文件，执行任务，并通过频道（如飞书/Discord）主动给用户发消息。
    - **收尾**：任务完成后立即删除 JSON 文件，且一次性 Cron 任务会自动从系统中移除。

### 2. 严谨性验证
通过“全静默测试”验证：在网关重启后，用户完全不发送任何消息，Agent 依然能准时在预定时间点弹出飞书消息。这证明了恢复逻辑不依赖于用户交互，而是由网关内部调度驱动。

## 适用场景
- 全局 Skill 安装后的自动重启通知。
- 修改端口、绑定地址等导致连接中断的配置变更。
- 任何需要 Agent “记得回来找我”的长时间运行任务。
