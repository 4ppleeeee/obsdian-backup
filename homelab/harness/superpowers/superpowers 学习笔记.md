 1. 通过 hook 的方式 在 session 开始时加入 

我先给你介绍一下我平时的工作（下面的序号不代表顺序）：
  1. 我是一个客户端开发，平时的需求分为 技术需求 和 业务需求 2类
  2. 针对技术需求，可能没有一个明确的tapd（一个内部的需求管理平台），但是我需要有一个明确的技术需求文档，描述清楚背景（收益）和具体做什么事儿
  3. 针对业务需求，产品经理提供完整的tapd（链接形式，可以通过mcp访问），设计师提供 figma 链接（可能是单独给到的，也可能是附在 tapd 的内容中）。
  关于要设计的这个 harness-engineering 架构，我有如下想法：

 4. 借鉴 openspec 的 workflow 模式，让整体流程有一个确定的起点
 5. 借鉴 openspec 的状态机模式，保证不会在没有明确的输出之前跳出循环。
 6. 借鉴 superpowers 的 hook 能力，能在执行到必要的阶段时保证插入某个重要的 skills
 7. 整个架构，希望可以设计多个 agent，分别代表：
		1. 产品经理（PM)——必须提供明确的 tapd 和 figma 文档，且 tapd 的内容和 figma 的内容可以相互验证（比如用 tapd mcp 获取的内容，理解后，和 figma 中的设计稿是否对应，是否包含了所有需要实现的场景），这一阶段，如果需求描述不清楚，必须停下来向人类来询问具体细节，禁止猜测，最终生成一份 prd.md
		2. 架构设计师(AD)，根据产品经理提供的最终版文档，列出来需要修改的点，其中要包括对当前所涉及的模块的架构评价，每次都给出 2 到 3 个方案，包括最小修改点的方案和如果存量架构不合理时顺带优化架构的方案，每个方案的优缺点，停下来询问人类采用哪种方案，最终生成一份方案文档 design.md
		3. 工程师(SDE)：根据架构师提供的方案文档，和人类的选择， 先列出 task.md（可以有多个，每个功能点一个）开始进行开发
		4. reviewer(RV)：针对代码进行 review（按照 rules），对代码是否实现了产品经理的文档中的需求进行 review
		5. 项目经理（PMO），作为中枢来调度上面 4 个 agent
8. 我预想的时，在整个流程的起点，有2个预设的命令： 
	1. tech-propose 处理技术需求的工作流，在这个工作流中 产品经理这个 agnet 就可以不需要，直接通过和用户的对话以及苏格拉底式提问，最终生成 架构设计师需要的 prd.md 即可
	2. business-propose 处理业务需求的工作流，在这个工作流中，四个 agent 都是必须的
