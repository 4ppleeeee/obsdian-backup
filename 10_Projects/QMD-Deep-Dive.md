# 写作素材：OpenClaw 语义记忆引擎 QMD 深度实战 📝

## 1. 痛点：AI 的“健忘症”与昂贵的 Token
*   **Token 消耗的三大来源**：
    1.  **系统背景 (System Prompt)**：每次对话都要带上的 `AGENTS.md`, `USER.md` 等基础设定。
    2.  **短期记忆 (Daily Logs)**：自动加载的今日与昨日日志，随着对话进行会迅速膨胀。
    3.  **长期记忆的挑战**：如果不加筛选地把所有历史都塞给 AI，Context Window 会瞬间爆炸，且每句话都在烧钱。
*   **解决方案**：引入 RAG (检索增强生成)，只把“最相关的几句话”喂给 AI。

## 2. 配置核心：`openclaw.json`
*   **传统模式**：默认使用内置 SQLite 索引，功能有限。
*   **QMD 模式**：
    ```json
    "memory": {
      "backend": "qmd",
      "qmd": {
        "command": "/home/aatroxli/.bun/bin/qmd",
        "paths": [{"path": "/path/to/workspace", "name": "workspace"}]
      }
    }
    ```

## 3. QMD 原理：本地化的“语义雷达”
*   **三位一体搜索**：
    1.  **BM25**：基于关键词的精确搜索。
    2.  **Vector (向量)**：基于语义理解的模糊搜索（即使没写关键词也能搜到）。
    3.  **Rerank (重排)**：对搜索结果进行二次精选，确保 AI 读到的是最准确的片段。

## 4. 安装进化史：从 Ollama 到“零配置”
*   **旧方案 (Ollama)**：
    *   需要安装 Ollama 服务，拉取 `nomic-embed-text` 模型。
    *   **优点**：统一管理，适合多应用共用。
    *   **缺点**：常驻内存（约 300MB+），对低配机器（如 4G 内存 VPS）不友好。
*   **新方案 (QMD 2.0.1+ 内置模型)**：
    *   安装环境：Bun (v1.3.10)。
    *   **特性**：首次运行自动下载 `embeddinggemma-300M-Q8_0.gguf`。
    *   **对比**：QMD 独立运行，按需加载。搜索时才唤醒模型，平时完全不占内存，完美适配轻量级 VPS。

## 5. 成果展示：速度与金钱的平衡
*   **Token 节省实验**：
    - 传统全量读取：消耗约 **6,500 Tokens**。
    - QMD 片段检索：消耗约 **450 Tokens**。
    - **结论**：Token 节省率高达 **93%**。
*   **反应时间实验 (CPU 模式)**：
    - **满血版 (含 Reranker/Expansion)**：由于需要加载 2GB 模型并进行 CPU 密集计算，耗时约 **341 秒** (5.6 分钟)。
    - **轻量版 (纯向量检索)**：仅使用 300MB 模型，耗时约 **1.5 秒**。
    - **结论**：在低配 VPS 上，关闭重排(Rerank)可以获得兼顾精准与速度的极佳体验。

## 6. 避坑指南：WebSocket 重复回答
*   **现象**：在使用高延迟检索（如满血版 QMD）时，飞书端频繁出现重复回答。
*   **排查**：确认为 WebSocket 连接断连重传导致。流式输出 (`blockStreaming: false`) 在网络波动时会加剧该现象。
*   **解决方案**：
    1. 减小上下文长度，提升检索速度。
    2. 开启 `blockStreaming: true`，利用块传输的 ACK 机制增强鲁棒性。
    3. 长期方案建议使用 Webhook 模式。
