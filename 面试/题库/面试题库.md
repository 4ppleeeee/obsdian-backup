## 第一阶段：Android 核心组件与语言特性

**关联点：** 简历中的“Android 客户端基础”、项目中的“登录/消息中心模块”。

### 1. Activity/Fragment 生命周期与状态管理

- **问题**：你在项目中实现了“停留时长埋点”，是在哪个生命周期计时和上报的？如果用户在聊天时突然接到电话，Activity 状态如何切换？
- **【标准答案】**：
    - **计时逻辑**：应在 `onResume()` 开始计时，在 `onPause()` 停止计时并记录时间差。因为 `onResume` 代表用户真正可见并可交互。
    - **状态切换**：接到电话时，Activity 经历 `onPause() -> onStop()`（Activity 进入后台但未销毁）。通话结束返回时，经历 `onRestart() -> onStart() -> onResume()`。
    - **项目延伸**：在 `TikTok_Message` 中，应注意在 `onSaveInstanceState()` 中保存当前输入框未发送的文本，防止内存回收导致的内容丢失。

### 2. Kotlin 核心语法特性

- **问题**：请谈谈 Kotlin 的 `lateinit var` 和 `by lazy` 在 Android 开发中的使用场景差异。
- **【标准答案】**：
    - **lateinit var**：延迟初始化。用于必须手动初始化的非空属性（如 `Dagger/Hilt` 注入、`onCreate` 中的组件绑定）。只能修饰 `var`。
    - **by lazy**：懒加载。用于属性首次调用时才初始化的场景。只能修饰 `val`，且默认线程安全。

## 第二阶段：架构设计与数据流

**关联点：** GitHub 项目中的“MVVM + Repository + SQLite”架构。

### 3. MVVM 与 LiveData 响应式 UI

- **问题**：为什么建议在 ViewModel 中持有 `MutableLiveData`，但在 View 层暴露只读的 `LiveData`？
- **【标准答案】**：为了遵循**单一职责原则**。ViewModel 拥有修改数据的权限，而 View（Activity/Fragment）只应负责观察数据变化，防止 View 层越权修改数据模型，保证数据流向的单向性。

### 4. 数据库升级与迁移 (Migration)

- **问题**：SQLite 的 `onUpgrade` 逻辑中，如果版本跨度较大，你是如何处理的？
- **【标准答案】**：采用**漏斗式/顺序升级逻辑**。使用 `if (oldVersion < 7) { ... } if (oldVersion < 8) { ... }` 确保无论用户从哪个旧版本更新，都能按顺序执行每一版的新增字段操作，保证数据完整性。

## 第三阶段：性能优化与工程实践

**关联点：** 简历中的“Handler 机制、多线程”、项目中的“数据看板、弱网体验”。

### 5. RecyclerView 极致优化

- **问题**：每 5 秒自动插入新消息时，如何避免 UI 闪烁？谈谈 `DiffUtil` 的作用。
- **【标准答案】**：使用 `DiffUtil` 进行局部刷新。它基于差分算法计算新旧两个 List 的差异，仅执行必要的 Item 更新（如 `notifyItemInserted`），避免 `notifyDataSetChanged()` 导致的全局重绘和闪烁。

### 6. Handler 机制与内存泄漏

- **问题**：`Handler`、`Looper`、`MessageQueue` 之间是什么关系？如何规避内存泄漏？
- **【标准答案】**：
    - **关系**：`Handler` 发送消息到 `MessageQueue`，`Looper` 负责死循环从队列捞取消息分发。
    - **方案**：1. 使用**静态内部类 + 弱引用 (WeakReference)** 持有 Activity；2. 在 `onDestroy` 中清空 Handler 消息队列（`removeCallbacksAndMessages(null)`）。

## 第四阶段：AI 与大模型深度思考 (The "Smart Test")

**关联点：** 简历中的“RAG 知识库构建”、“表格大模型微调”、“Vibe Coding”。

### 7. 落地策略：RAG vs Fine-tuning

- **问题**：如果你要为 `TikTok_Message` 开发一个能回答“我上周和谁聊了关于‘面试’的话题”的智能助手，你会选择 Fine-tune 还是 RAG？
- **【标准答案】**：**必须选择 RAG**。1. **时效性**：消息秒级更新，RAG 可实时检索最新聊天记录；2. **隐私性**：私密数据不应留在模型权重中；3. **真实性**：对于事实性数据，Fine-tune 模型易产生幻觉，RAG 可提供证据。

### 8. 端侧 AI 的权衡 (On-Device AI)

- **问题**：如果在 Android 手机本地部署一个 3B 参数的模型来实现“消息自动摘要”，你会最关注哪三个技术指标？
- **【标准答案】**：1. **首字延迟 (TTFT)**（交互体验）；2. **内存占用与功耗**（量化后的 RAM 占用及发热控制）；3. **Token 吞吐量**（长会话处理速度）。

### 9. 范式理解：Vibe Coding 的本质

- **问题**：如果 AI 已经能完成 90% 的业务代码编写，你认为未来 3 年内，一名顶尖 Android 开发者的核心竞争力将如何转型？
- **【标准答案】**：核心竞争力将转向：1. **复杂架构设计与系统底层逻辑决策**；2. **需求到 Prompt 的精准建模**；3. **AI 无法触及的深度性能调优（如跨进程 Binder 优化）**。
