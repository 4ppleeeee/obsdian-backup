在 Android 开发中，单例模式（Singleton Pattern）是最常用的设计模式之一。虽然看似简单，但在多线程并发、内存泄漏以及 Android 特有的生命周期环境下，单例的实现有很多讲究。

以下是单例模式的几种主要实现形式及其深度解析：

---

## 1. 饿汉式（Static Constant）

在类加载时就完成了初始化。

Kotlin

```
object EagerSingleton {
    // Kotlin 的 object 关键字原生支持饿汉式单例
    fun doSomething() {}
}
```

- **原理：** 利用 ClassLoader 机制保证初始化时只有一个线程。
    
- **优点：** 线程绝对安全，实现最简单。
    
- **缺点：** **非懒加载**。如果该单例非常占内存且程序运行期间没用到，会造成资源浪费。
    

---

## 2. 懒汉式（线程不安全）

只在第一次用到时才创建。

Java

```
public class LazySingleton {
    private static LazySingleton instance;
    private LazySingleton() {}

    public static LazySingleton getInstance() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }
}
```

- **风险：** 多线程下，两个线程同时判断 `instance == null`，会导致创建两个实例。**生产环境禁用**。
    

---

## 3. 双重检查锁定（DCL - Double Check Locking）

这是 Android 开发中最推荐的 Java 实现方案。

Java

```
public class DclSingleton {
    // 必须加 volatile，防止指令重排
    private static volatile DclSingleton instance;
    private DclSingleton() {}

    public static DclSingleton getInstance() {
        if (instance == null) { // 第一次检查：避免不必要的同步
            synchronized (DclSingleton.class) {
                if (instance == null) { // 第二次检查：确定真的没被创建
                    instance = new DclSingleton();
                }
            }
        }
        return instance;
    }
}
```

- **核心知识点：**
    
    - **第一个 `if`：** 为了性能。如果单例已创建，直接返回，不用进 `synchronized`（避免昂贵的锁开销）。
        
    - **第二个 `if`：** 为了安全。防止 Thread A 进锁创建完后，Thread B 刚好醒来再次创建。
        
    - **`volatile`：** 极其重要。防止 `new` 对象时指令重排导致 Thread B 拿到一个**未初始化完成**的半成品。
        

---

## 4. 静态内部类（Holder）

利用 JVM 的类加载特性实现的“懒加载+线程安全”方案。

Java

```
public class HolderSingleton {
    private HolderSingleton() {}
    
    public static HolderSingleton getInstance() {
        return Holder.INSTANCE;
    }

    private static class Holder {
        private static final HolderSingleton INSTANCE = new HolderSingleton();
    }
}
```

- **原理：** `Holder` 类只有在 `getInstance()` 被调用时才会加载，类加载过程是线程安全的。
    
- **优点：** 兼顾了懒加载和高性能（无锁）。
    

---

## 5. 枚举单例（Enum）

《Effective Java》作者极力推荐的方案。

Java

```
public enum EnumSingleton {
    INSTANCE;
    public void doSomething() {}
}
```

- **优点：** **绝对安全**。不仅线程安全，还能防止利用“反射”和“序列化”破坏单例（这是其他方案都无法天然防范的）。
    

---

## 6. Android 特有的单例陷阱：Context 内存泄漏

在 Android 中实现单例时，如果需要传 `Context`，必须极其小心。

Kotlin

```
// 错误案例：会引发内存泄漏
class LeakManager private constructor(private val context: Context) {
    companion object {
        @Volatile private var instance: LeakManager? = null
        fun getInstance(context: Context) = instance ?: synchronized(this) {
            instance ?: LeakManager(context).also { instance = it }
        }
    }
}
```

- **危险点：** 如果你传入的是 `Activity` 的 Context，单例（静态变量）会一直持有该 Activity 的引用，导致 Activity 销毁后内存无法回收。
    
- **正确做法：** 始终使用 **`context.applicationContext`**。

## 7. 总结与面试指南

作为面试官，你可以按照这个逻辑去考察：

1. **入门：** “手写一个 DCL 单例。”
    
2. **进阶：** “为什么 DCL 里要加 `volatile`？”（考察指令重排）。
    
3. **深度：** “静态内部类和 DCL 哪个更好？”（考察对类加载时机和锁开销的理解）。
    
4. **Android 场景：** “单例中持有 Context 怎么优化？”（考察内存泄漏意识）。
    

**面试官彩蛋：**

你可以追问：“单例在多进程环境下还是单例吗？”

（_答案：_ 每一个进程都有独立的 JVM 实例，所以**多进程下单例失效**。需要通过 ContentProvider 或 AIDL 进行跨进程通信）。
