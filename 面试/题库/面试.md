## Android 基础（20 道题）

1. 【问题】（初级）
从简历“掌握 Activity/Fragment/Service”等组件延伸：请说出 Activity 的完整生命周期，并说明从 A 页面跳转到 B 页面再返回时，A 的回调顺序。

【答案】：
Activity 常见生命周期是：`onCreate -> onStart -> onResume`（进入前台），离开前台时会触发 `onPause`，若完全不可见再触发 `onStop`，销毁时触发 `onDestroy`。从 A 跳到 B 时，A 通常先 `onPause`，如果 B 是全屏，A 继续 `onStop`。从 B 返回 A 时，A 会走 `onRestart -> onStart -> onResume`。如果 A 进程被系统回收，则返回时会重新创建，重新走 `onCreate`。

2. 【问题】（中级）
从消息列表项目“页面状态恢复”延伸：`onSaveInstanceState` 与 `ViewModel` 的使用边界是什么？

【答案】：
`onSaveInstanceState` 适合保存轻量、可序列化、与 UI 恢复强相关的数据，例如滚动位置、输入框文本、选中 Tab。`ViewModel` 适合保存业务状态和异步结果，例如消息列表数据、分页状态、加载状态。两者应配合使用：`ViewModel` 扛业务状态，`Bundle` 兜底关键 UI 状态，这样在旋转屏幕和进程重建时都更稳。

3. 【问题】（初级）
从“Android 基础能力”延伸：四大组件分别是什么、职责是什么？

【答案】：
四大组件是 `Activity`、`Service`、`BroadcastReceiver`、`ContentProvider`。`Activity` 负责界面交互；`Service` 负责后台任务或长期运行能力；`BroadcastReceiver` 负责接收广播事件；`ContentProvider` 负责结构化数据共享和跨进程访问。工程上要避免把所有逻辑都放进 Activity，应按组件职责拆分。

4. 【问题】（中级）
从消息系统“通知点击跳转会话页”延伸：显式 Intent、隐式 Intent、PendingIntent 的区别与风险点？

【答案】：
显式 Intent 指定目标组件，路径明确、可控；隐式 Intent 通过 action/category/data 匹配组件，适合开放能力。`PendingIntent` 是把未来执行某个 Intent 的能力交给系统或第三方，需注意不可变标记（`FLAG_IMMUTABLE`）与更新策略（`FLAG_UPDATE_CURRENT`），否则可能出现 Intent 被替换或复用导致安全问题。

5. 【问题】（中级）
从简历“熟悉 Fragment”延伸：Fragment 生命周期与 View 生命周期有何区别？

【答案】：
Fragment 本体生命周期和其 View 生命周期并不完全一致。`onCreateView/onViewCreated` 到 `onDestroyView` 是 View 生命周期，适合做 ViewBinding 绑定/解绑。Fragment 本身可能还活着，但 View 已销毁；若此时继续持有 View 引用容易内存泄漏，因此应在 `onDestroyView` 释放 UI 相关引用。

6. 【问题】（中级）
从“会话列表 Fragment 与聊天详情 Fragment”延伸：Fragment 间推荐怎样通信？

【答案】：
推荐使用同一 Activity 作用域的共享 `ViewModel` 传递状态，或使用 `FragmentResult` 传一次性结果。尽量避免 Fragment 互相直接持有引用和调用，降低耦合，减少生命周期错配导致的崩溃和空指针。

7. 【问题】（中级）
从简历“RecyclerView 列表展示”延伸：为什么不建议频繁 `notifyDataSetChanged`？

【答案】：
`notifyDataSetChanged` 会触发全量刷新，丢失细粒度变更信息，导致不必要重绘和动画抖动。更推荐 `DiffUtil/ListAdapter` 计算新旧差异，执行最小化更新，性能更好、视觉更平滑，尤其在消息会话这种高频更新列表里收益明显。

8. 【问题】（中级）
从项目“分页加载会话列表”延伸：分页设计中有哪些关键点？

【答案】：
要点包括：页大小设计、触底预加载阈值、防重复请求、请求取消与串行控制、加载态与错误态统一管理。渲染侧要减少 `onBind` 重计算，利用局部刷新，必要时用 Paging3 管理分页状态。目标是首屏快、滚动稳、弱网可恢复。

9. 【问题】（高级）
从“会话已读未读状态更新”延伸：如何使用 payload 做局部刷新？

【答案】：
通过 `notifyItemChanged(position, payload)` 只传变更字段，在 `onBindViewHolder(holder, position, payloads)` 中判断 payload 并仅更新对应控件，例如红点、时间、未读数。这样避免整项重绑，减少闪烁和布局重算。

10. 【问题】（初级）
从岗位要求“androidx”延伸：AndroidX 相比旧 support 库的价值是什么？

【答案】：
AndroidX 统一包名 `androidx.*`，与 Jetpack 深度协同，版本迭代更独立。它改善了兼容性与维护性，便于接入 Lifecycle、Room、Navigation、WorkManager 等现代组件。迁移过程中可借助 Jetifier 过渡，但长期应清理旧依赖。

11. 【问题】（中级）
从项目“MVVM + Repository + LiveData”延伸：请描述完整数据流。

【答案】：
UI 层发起用户意图（刷新、翻页、发送消息），`ViewModel` 接收后调用 `Repository`，Repository 聚合本地数据库和远端接口结果，最终通过 `LiveData/StateFlow` 暴露状态给 UI。UI 只负责渲染状态，不直接访问数据源，实现职责清晰、可测试。

12. 【问题】（中级）
从简历“LiveData 使用经验”延伸：LiveData 与 StateFlow 的主要区别？

【答案】：
LiveData 自带生命周期感知，适合传统 XML 页面；StateFlow 属于协程体系，可组合性更强，适合复杂状态流。LiveData 在旧项目接入成本低，StateFlow 更适合新项目架构统一。工程上常见方案是 ViewModel 内部用 Flow，向 UI 暴露 StateFlow。

13. 【问题】（高级）
从岗位要求“MVI、MVVM”延伸：MVI 与 MVVM 的核心差异是什么？

【答案】：
MVI 更强调单向数据流与不可变状态，所有状态变化通过 Intent -> Reducer -> State，利于回放与调试。MVVM 更灵活，接入成本低，但在复杂页面可能出现状态分散和时序难追踪问题。复杂交互场景下 MVI 可读性和可维护性通常更好。

14. 【问题】（高级）
从消息项目延伸：若将当前 MVVM 改造成 MVI，你会如何拆分？

【答案】：
先定义 `Intent`（加载、刷新、发送、重试）、`State`（数据、分页游标、loading、error、会话状态）、`Effect`（Toast、导航、一次性提示）。ViewModel 作为状态机处理 Intent，Reducer 产出新 State，UI 只消费 State。一次性事件单独用 Effect，避免状态污染与重复消费。

15. 【问题】（中级）
从简历“SQLite 查询性能优化、索引设计”延伸：联合索引设计原则是什么？

【答案】：
联合索引应围绕高频查询条件和排序字段设计，遵循最左前缀原则。并非索引越多越好，索引会增加写入成本和存储开销。应通过 `EXPLAIN QUERY PLAN` 验证是否命中，结合业务读写比进行权衡。

16. 【问题】（中级）
从项目“使用 Handler 机制优化”延伸：Handler 造成内存泄漏的典型原因和规避方式？

【答案】：
非静态内部类 Handler 默认持有外部类引用，延迟消息未清理会导致页面退出后仍被引用。规避方式：静态内部 Handler + 弱引用，页面销毁时执行 `removeCallbacksAndMessages(null)`；或者使用协程 + Lifecycle 感知机制减少显式 Handler 使用。

17. 【问题】（中级）
从“理解消息循环”延伸：Looper、MessageQueue、Handler 三者关系？

【答案】：
Looper 负责循环取消息，MessageQueue 存消息，Handler 负责发消息和处理消息。主线程 Looper 不停消费消息驱动 UI 事件。耗时任务若占用主线程会阻塞消息处理，导致卡顿甚至 ANR。

18. 【问题】（中级）
从岗位要求“跨端”延伸：Android 与 H5 混合开发中 JSBridge 的常见风险和防护？

【答案】：
风险包括接口暴露过宽、参数未校验、域名未白名单、可被注入恶意脚本调用。防护要点：域名白名单、能力最小化、参数签名/校验、桥接调用审计日志、敏感能力二次确认。目标是既可用又可控。

19. 【问题】（中级）
从岗位要求“开源库”延伸：评估开源库时你看哪些维度？

【答案】：
核心看稳定性（维护活跃度、issue 响应）、性能（内存、CPU、包体）、兼容性（AndroidX/协程/Compose）、安全合规（许可证）、可观测性（日志埋点能力）和迁移成本。避免只看 star 数，重点看工程可持续性。

20. 【问题】（高级）
从项目“稳定性提升”延伸：如何搭建 Android 稳定性治理闭环？

【答案】：
先定义指标：崩溃率、ANR、卡顿、启动时长；再做分层监控（UI、网络、数据库、线程池）；建立灰度与回滚机制；对高频问题做专项治理和复盘模板；最终沉淀自动化回归与守护规则，实现“发现-定位-修复-防复发”的闭环。

## Java 基础（20 道题）

1. 【问题】（初级）
从简历“了解 JVM 基本原理”延伸：JVM 运行时内存区域有哪些？

【答案】：
主要包括程序计数器、虚拟机栈、本地方法栈、堆、方法区（JDK8 以后以元空间实现）。对象通常分配在堆，方法调用在栈帧中执行。理解内存区域有助于排查 OOM、栈溢出和类元数据问题。

2. 【问题】（中级）
从 JVM 知识延伸：类加载过程和双亲委派机制是什么？

【答案】：
类加载一般经历加载、验证、准备、解析、初始化。双亲委派是“先委派父加载器，父加载失败再自己加载”，可避免核心类被重复加载和篡改。插件化场景可打破委派，但要控制隔离边界。

3. 【问题】（中级）
从并发基础延伸：`volatile` 能保证什么，不能保证什么？

【答案】：
`volatile` 保证可见性和一定的有序性（禁止特定重排），不能保证复合操作原子性，比如 `count++` 仍然不安全。适用于状态位、开关变量等场景，不适合替代锁处理复杂并发修改。

4. 【问题】（中级）
从“锁机制”延伸：`synchronized` 的底层原理是什么？

【答案】：
`synchronized` 基于对象监视器（Monitor）实现，可重入。JVM 会根据竞争程度做锁优化和升级。它语义简单、可读性高，适用于大多数互斥场景。

5. 【问题】（中级）
从并发实战延伸：`ReentrantLock` 与 `synchronized` 如何选型？

【答案】：
如果只需基础互斥，`synchronized` 更简洁。若需要可中断、超时获取、公平策略、多条件队列，`ReentrantLock` 更灵活。选型应基于业务需求而不是“谁更高级”。

6. 【问题】（高级）
从锁原理延伸：什么是 AQS？

【答案】：
AQS（AbstractQueuedSynchronizer）是并发同步器基础框架，使用 `state` 表示同步状态，竞争失败线程进入队列等待。`ReentrantLock`、`CountDownLatch`、`Semaphore` 都基于 AQS 构建。

7. 【问题】（中级）
从“锁 + 幂等判断”延伸：什么是 CAS？什么是 ABA 问题？

【答案】：
CAS 通过“比较并交换”实现无锁原子更新。ABA 问题是值从 A 变 B 又变回 A，CAS 看起来没变化却忽略了中间修改。可通过版本号（如 `AtomicStampedReference`）或业务幂等键规避。

8. 【问题】（中级）
从 JVM/GC 延伸：Java 如何判断对象是否可回收？

【答案】：
主流 JVM 使用可达性分析，从 GC Roots 出发不可达对象视为可回收。并不是单纯引用计数，因为循环引用会误判。这个机制是理解 GC 行为和内存泄漏定位的基础。

9. 【问题】（中级）
从 GC 基础延伸：Minor GC 和 Full GC 区别是什么？

【答案】：
Minor GC 主要回收新生代，通常更频繁、停顿较短；Full GC 会涉及老年代甚至元空间，停顿更重，对线上延迟影响大。实际优化重点是减少不必要 Full GC 触发。

10. 【问题】（高级）
从性能排查延伸：你会看哪些 GC 指标判断系统是否健康？

【答案】：
关注 GC 次数、停顿时间（尤其 P99）、对象分配速率、晋升速率、老年代占用趋势。结合 GC 日志可判断是短期分配高峰、长期泄漏，还是参数设置不合理。

11. 【问题】（中级）
从数据结构基础延伸：`HashMap` 的 put/get 过程简述。

【答案】：
先计算 hash 定位桶位，冲突时在桶内链表或红黑树查找；`put` 时若 key 已存在则覆盖，否则插入。容量阈值达到会触发扩容并迁移。理解冲突与扩容是分析性能问题的关键。

12. 【问题】（中级）
从并发集合延伸：`ConcurrentHashMap` 为什么比 `Collections.synchronizedMap` 更高效？

【答案】：
`ConcurrentHashMap` 采用更细粒度并发控制，读操作大多无锁，写操作局部加锁或 CAS，减少全局锁竞争。`synchronizedMap` 本质是整表互斥，吞吐量在高并发下受限。

13. 【问题】（初级）
从集合使用延伸：`ArrayList` 与 `LinkedList` 的适用场景？

【答案】：
`ArrayList` 随机访问快、内存连续，适合读多写少。`LinkedList` 插删节点理论上快，但查找慢且缓存不友好。多数业务场景优先 `ArrayList`，除非确实频繁在已知位置插删。

14. 【问题】（中级）
从有序结构延伸：`TreeMap` 的实现和时间复杂度是什么？

【答案】：
`TreeMap` 基于红黑树实现，增删查平均为 `O(log n)`，支持有序遍历和范围查询。若不需要顺序语义，`HashMap` 通常性能更好。

15. 【问题】（中级）
从工程实践延伸：线程池参数如何配置？常见错误是什么？

【答案】：
线程池配置要结合任务类型（CPU 密集/IO 密集）和系统负载，重点是核心线程数、最大线程数、队列长度、拒绝策略。常见错误是无界队列导致任务堆积、延迟不可控，或默认拒绝策略未做业务兜底。

16. 【问题】（中级）
从线上排查延伸：常见 OOM 类型有哪些？如何初步定位？

【答案】：
常见有 `Java heap space`、`Metaspace`、`Direct buffer memory`。定位时先看错误类型，再结合堆转储、对象直方图、线程与流量变化，判断是泄漏、缓存失控还是瞬时流量冲击。

17. 【问题】（中级）
从并发问题延伸：什么是死锁？如何规避？

【答案】：
死锁是多个线程互相等待彼此持有资源，形成闭环。规避方式包括固定加锁顺序、减少锁嵌套、使用超时锁、尽量细化锁粒度，并配合线程 dump 快速定位。

18. 【问题】（初级）
从字符串基础延伸：为什么 `String` 设计为不可变？

【答案】：
不可变带来线程安全、哈希稳定和常量池复用优势，适合作为 Map key 与共享对象。代价是频繁拼接产生中间对象，因此大量拼接应使用 `StringBuilder`。

19. 【问题】（中级）
从集合正确性延伸：为什么重写 `equals` 时通常必须重写 `hashCode`？

【答案】：
哈希容器先按 `hashCode` 定位桶，再用 `equals` 判断相等。若 `equals` 相等但 `hashCode` 不一致，会出现逻辑相等对象无法命中查询的问题，破坏集合语义。

20. 【问题】（高级）
从简历“并发保护机制设计”延伸：请设计一个防止重复提交的通用方案。

【答案】：
可采用“请求幂等键 + 状态机 + 分布式锁/原子操作”组合。请求携带唯一幂等键，服务端先查键是否处理过；处理中状态要有超时和重试策略；最终态需可回放可追踪。关键不是只加锁，而是保证跨重试、跨实例的一致性。

## AI 相关（建议题）

1. 【问题】
你在简历里提到 Transformer、Diffusion、Agent。请用通俗语言解释 Attention 的作用。

【参考答案】：
Attention 的核心是“让模型在处理当前词时，动态关注输入中最相关的其他词”。它能更好建模长距离依赖，且可并行训练，比传统 RNN 更适合大规模文本建模。

2. 【问题】
什么是 RAG？和 Fine-tuning 的边界在哪里？

【参考答案】：
RAG 是“先检索再生成”，适合知识更新频繁、要求可引用来源的场景；Fine-tuning 是修改模型参数，适合固定任务模式和长期稳定风格。工程上常先用 RAG 快速落地，再按稳定需求考虑微调。

3. 【问题】
你如何定义“大模型幻觉”？如何降低？

【参考答案】：
幻觉是模型输出看似合理但不符合事实或无依据的内容。降低方式包括高质量检索、提示词约束、结构化输出、结果校验和关键场景人工复核。

4. 【问题】
什么是 Embedding？向量数据库为什么能做语义检索？

【参考答案】：
Embedding 把文本映射为高维向量，语义相近文本在向量空间距离更近。向量数据库通过近邻搜索找到语义相似内容，因此能支持“意思相近但关键词不同”的检索。

5. 【问题】
Agent 和普通问答模型的关键区别是什么？

【参考答案】：
普通问答偏单轮生成；Agent 具备规划和多步执行能力，可调用外部工具、迭代决策并产出任务结果。它更像“任务执行者”，而不是只会回答问题。

6. 【问题】
如果让你在 Android 项目里用 AI 工具，你会怎么用？

【参考答案】：
可用于样板代码、单测草稿、日志初筛、文档生成和重构建议。高风险逻辑（权限、并发、支付链路）必须人工评审与测试验证，避免“盲信 AI 生成代码”。

7. 【问题】
你如何看待“AI 会不会替代初级开发”？

【参考答案】：
AI 会替代重复编码，但不会替代工程判断和责任闭环。开发者价值会从“写代码”转向“定义问题、约束方案、验证质量、保障上线稳定”。

8. 【问题】
团队引入 AI 编码助手时，最重要的 3 条规范是什么？

【参考答案】：
第一，数据安全：禁止上传敏感代码和密钥；第二，质量门禁：AI 生成代码必须过测试和 Code Review；第三，可追溯性：保留关键提示词与改动依据，便于审计与复盘。

9. 【问题】
Temperature、Top-p、System Prompt 分别影响什么？

【参考答案】：
Temperature 控制输出随机性，Top-p 控制采样候选范围，System Prompt 约束模型行为边界。生产场景通常降低随机性，提高稳定性和可复现性。

10. 【问题】
你如何评估一个 AI 功能是否可上线？

【参考答案】：
要同时看离线效果（准确率、召回率、幻觉率）和线上指标（时延、转化、投诉率），并准备灰度、回滚、人工兜底和监控告警。可上线不等于模型分高，而是系统整体可控。
